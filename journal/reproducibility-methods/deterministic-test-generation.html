Deterministic Test Generation via Seed-Controlled Stochastic AST Synthesis

Freddy Medina
ItsNotAI LABS Research Division
June 2026

Abstract

Reproducibility in AI benchmarking fails most often at the point where test data are generated. When benchmark operators cannot regenerate the exact evaluation corpus used for a published result, claims of robustness, correctness, or speed become observationally fragile. This paper presents a deterministic methodology for stochastic abstract syntax tree (AST) synthesis in which all randomness is derived from a single seeded pseudorandom number generator, yielding bit-for-bit identical datasets when configuration and implementation are held fixed. We formalize the FableBreaker generator as a transition system driven by random.Random(seed), characterize node-selection distributions through explicit parameter vectors, and show how maximum depth constraints, per-depth decay factors, and controlled adversarial family injection can be combined without sacrificing determinism. The resulting construction preserves the coverage advantages of stochastic synthesis while eliminating ambiguity in corpus regeneration. We provide an algorithmic specification, discuss implementation details including canonical JSON serialization and elimination of hidden entropy sources, and state a proof that the generator is a pure function of its seed and fixed parameters. Cross-platform experiments across multiple CPython releases and operating systems produce identical dataset hashes, supporting the claim that deterministic dataset equivalence is achievable in practice. The methodology is directly applicable to benchmark governance, certification, and audit workflows where exact replay of public and hidden test suites is a prerequisite for scientific trust.

Keywords: reproducibility, deterministic generation, AST synthesis, pseudorandomness, benchmark governance, dataset equivalence, adversarial testing

1. Introduction

The reproducibility crisis in empirical machine learning is often discussed in terms of model initialization, training nondeterminism, or unavailable datasets. Benchmark construction itself is less frequently examined, despite the fact that benchmark outcomes are only as stable as the test corpora that instantiate them. In adversarial evaluation settings, the benchmark generator is not a passive utility. It is a scientific instrument that defines the operational distribution against which candidate systems are judged. If two laboratories using the same nominal benchmark version synthesize different instances because a hidden entropy source leaks into the generator, then any comparison of system performance becomes confounded by dataset drift rather than reflecting differences in algorithmic quality.

FableBreaker addresses this problem by treating test generation as a deterministic computation parameterized by a seed. The approach is deliberately austere: one pseudorandom number generator is created from one integer seed, and every stochastic decision made during corpus construction must be a measurable consequence of draws from that generator. No wall-clock time, process identifier, hash randomization salt, thread interleaving, or operating-system dependent directory ordering may affect the output. The gain is substantial. Once a benchmark operator publishes a seed and fixed configuration, any verifier can reconstruct the precise dataset, recompute expected outputs, and audit the fairness of the scoring process.

The central tension is that stochastic synthesis is valuable. Randomized test generation broadens structural coverage, reveals edge-case interactions, and prevents overfitting to a narrow hand-written suite. The objective is therefore not to remove stochasticity, but to encapsulate it. This paper shows that stochastic generation and reproducibility are compatible when randomness is modeled as an explicit, replayable input rather than an ambient property of execution.

Our contributions are fourfold. First, we define deterministic dataset equivalence and specify a seed-controlled stochastic AST generator consistent with that definition. Second, we describe parameterized node probabilities, depth control, and adversarial family injection under a single-source-of-stochasticity discipline. Third, we present a proof that the generator is a pure function of the seed and fixed configuration. Fourth, we report cross-platform reproducibility results showing identical dataset digests across CPython versions and operating systems.

2. Reproducibility Failure Modes in AI Benchmarking

Benchmark irreproducibility can arise from mundane implementation details. The first class of failures stems from hidden entropy. Calls to random.random() on the module-level singleton, implicit sampling in third-party libraries, or dependence on iteration order over data structures populated from nondeterministic sources all create output variance that is difficult to observe locally yet devastating for public verification. The second class stems from partial specification. Papers may report only a benchmark version identifier while omitting seed values, generator parameters, or split-selection policies. The third class stems from serialization ambiguity, such as using JSON encoders that do not enforce key ordering or text encodings that vary across platforms.

These failures are magnified in AI benchmarking because benchmark instances are often generated rather than stored exhaustively. A generator that produces semantically rich programs, prompts, tasks, or evaluation traces occupies less storage and offers broader coverage than a fixed file archive, but only if it can be rerun exactly. Without exact regeneration, benchmarking ceases to be falsifiable in the Popperian sense. Researchers may publish a performance claim, yet auditors cannot determine whether the same system evaluated on the same data would reproduce the result.

Adversarial benchmarks are especially sensitive because the distribution of test cases is itself part of the research contribution. FableBreaker uses families such as dynamic_match_storm, del_erasure_trap, duplication_aliasing, branch_balance, deep_pair_projection, and modular_arithmetic_net. Each family targets a distinct semantic weakness in candidate evaluators. If family frequencies drift due to uncontrolled randomness, then the benchmark no longer measures the intended stress profile.

We therefore adopt the position that reproducibility is not an after-the-fact property but a primary design constraint. The generator must expose all degrees of freedom explicitly and consume them in a fixed order.

3. Deterministic Synthesis Model

3.1 Generator state and single-source stochasticity

Let S be the set of integer seeds, Θ the set of fixed generator configurations, and D the set of serialized datasets. A generator is a mapping G : S × Θ → D. Determinism requires that for any fixed configuration θ ∈ Θ, repeated executions of G(seed, θ) yield the same serialized dataset byte sequence.

In the FableBreaker architecture, the state transition begins with rng = random.Random(seed). This object is the only authorized source of stochasticity. Every case constructor receives rng explicitly as an argument. The reference implementation avoids module-level random state and avoids constructing additional generators inside family builders. Because Python objects are passed by reference, advancing rng in one builder deterministically affects the next builder's draws, producing a total order over all random events.

The existing generator implementation is illustrative. The top-level main function constructs a single random.Random(args.seed) instance, iterates over case indices in ascending order, and writes line-delimited JSON with sort_keys=True. The make_case routine consumes randomness for family selection and family-specific parameters such as sizes, tags, or modular bases. This design is sufficient to state a determinism theorem because no branch of execution queries any source of entropy other than rng.

3.2 Formal definitions

Definition 1 (Stochastic trace).
For fixed seed s and configuration θ, the stochastic trace T(s, θ) is the ordered sequence of pseudorandom draws consumed during execution of G(s, θ).
Definition 2 (Deterministic dataset equivalence).
Two executions are dataset-equivalent if their serialized outputs are byte-identical. We write D₁ ≡ D₂ iff sha256(D₁) = sha256(D₂) and the byte streams match positionally.
Definition 3 (Pure seeded generator).
A generator G is pure with respect to seed under configuration θ if for all seeds s, G(s, θ) depends only on s and θ, and has no observational dependence on mutable external state.

Definition 2 is intentionally stronger than semantic equivalence. A corpus whose cases are merely isomorphic is insufficient for auditability. Reproducibility workflows need exact byte-level identity so that manifests, expected hashes, and certification records remain stable across institutions.

3.3 Parameterized node distributions

General stochastic AST synthesis is usefully described through node-type distributions. Let N be the set of node constructors, such as literals, binary operations, pairs, tags, matches, erasures, duplications, and iterative forms like repeat. At depth d, the probability of selecting node type n is defined as

P(n | d, θ) = exp(αₙ - λₙ d) / Σₘ exp(αₘ - λₘ d)

where αₙ is a base log-weight and λₙ is a depth decay coefficient. This parameterization has two desirable properties. First, it is explicit and versionable: benchmark papers can publish the exact vector (α, λ). Second, it induces a controlled contraction toward terminal nodes as depth increases, reducing the probability of pathological recursion while preserving variety near the root.

The current FableBreaker generator uses family-specific constructors rather than a single grammar-wide sampler, but the same principle applies. Within a family, draws choose tags, operations, sizes, or projection paths. Families may themselves be treated as high-level nonterminals with probabilities ρ_f. Thus the overall probability of a case can be decomposed into P(family | θ) × Π_t P(draw_t | prefix_t, θ). Determinism is unaffected by the mathematical form of these distributions so long as all samples are read from the same pseudorandom trace.

4. Depth Control and Adversarial Family Injection

4.1 Depth control mechanisms

Depth control is essential for both computational tractability and distributional transparency. A generator that recursively expands ASTs without explicit depth governance may terminate only in expectation, which is unacceptable for certified benchmarks. We employ a two-part scheme. First, max_depth provides a hard cap: when the remaining depth budget is zero, the sampler is restricted to terminal or depth-neutral constructors. Second, decay factors reduce the probability of choosing expansive nodes before the cap is reached. This yields smoother shape distributions than a hard cut alone.

Although several reference families in the current code are built iteratively rather than recursively, the same discipline appears in practice. Case size parameters bound the number of loop iterations, and derived statistics record reference_max_depth from evaluation. In generalized synthesis, we recommend publishing both the target generation ceiling and the observed evaluation-depth distribution. The two are not identical: a shallow syntactic tree can induce deeper evaluation through constructs like repeat.

A practical parameterization is

effective_weightₙ(d) = base_weightₙ × decayₙ^d,  0 < decayₙ ≤ 1.

Terminal nodes receive decay near one, while branching nodes receive smaller decay. At d = max_depth, all nonterminal weights are set to zero. Because the depth and all weights are deterministic functions of preceding draws and configuration, the introduction of depth control does not create any new entropy source.

4.2 Controlled adversarial family injection

Adversarial families are not sampling accidents; they are policy instruments. Let F be the finite set of families and ρ_f a rational injection rate for family f, with Σ_f ρ_f = 1. Controlled injection means the benchmark operator can either sample families iid according to ρ using the seeded generator, or construct a deterministic schedule whose empirical frequencies converge exactly to ρ over count cases. The second option is useful when strict family balance is required.

FableBreaker's current implementation samples from a fixed list of six families using the seeded generator. This already guarantees that a given seed induces the same family sequence everywhere. For certification-oriented suites, we further recommend a mixed design: deterministic quotas at the batch level, seeded shuffling within each batch, and family-local stochasticity within each case. Such a design preserves diversity while making the adversarial profile auditable.

5. Algorithmic Specification

The deterministic generator can be written as a pure state transformer over the pseudorandom state. Algorithm 1 gives a reference specification compatible with the FableBreaker implementation strategy.

Algorithm 1: Seed-controlled deterministic dataset synthesis
Input: seed s, configuration θ = (count, split, max_depth, α, λ, ρ, serializer)
Output: line-delimited dataset D

1: rng ← Random(s)
2: D ← []
3: for i in 0 .. count-1 do
4:     f ← sample_family(rng, ρ, i, split)
5:     size ← sample_size(rng, split, f)
6:     expr ← build_case(rng, f, size, max_depth, α, λ)
7:     value, stats ← evaluate(expr)
8:     record ← {
9:         id: format_id(split, i, f),
10:        split: split,
11:        family: f,
12:        size_hint: size,
13:        expr: expr,
14:        expected_sha256: digest(value),
15:        reference_steps: stats.steps,
16:        reference_max_depth: stats.max_depth
17:    }
18:    D.append(serialize_sorted_json(record))
19: end for
20: return join_lines(D) + "\\n"

The specification enforces determinism at three levels. First, random draws are consumed in a fixed order. Second, record fields are assembled in a fixed schema. Third, serialization uses a canonical encoding, e.g., UTF-8 and lexicographically sorted keys. Importantly, evaluation of the generated expression is itself deterministic because the AST language uses integer arithmetic, ordered dictionaries only at serialization boundaries, and explicit semantics for erasure, duplication, matching, and bounded repetition.

6. Purity Theorem and Proof

Theorem 1. Fix a generator configuration θ including the family catalog, size ranges, depth policy, evaluator semantics, and serializer. Assume that every stochastic branch in the implementation consumes entropy exclusively by invoking methods on a single object rng = Random(seed), that no mutable external state is read, and that all helper routines are deterministic functions of their explicit inputs. Then the dataset generator G(seed, θ) is pure with respect to seed.

Proof. The proof proceeds by structural induction over the call graph of G. The base step is the initialization of rng from seed, which yields a deterministic internal state because the pseudorandom algorithm and seeding procedure are fixed by θ. For the induction hypothesis, assume that after k random draws the program state is a deterministic function of seed and θ. The (k+1)-st stochastic event is produced by applying a deterministic method of rng to that state, so its outcome is also a deterministic function of seed and θ. All nonstochastic helper routines, by assumption, preserve determinism under function composition. Therefore each case record is a deterministic function of seed and θ. The output dataset is a fixed-order concatenation of deterministic records under a deterministic serializer, hence is itself a deterministic function of seed and θ. No term in the construction depends on external mutable state. Therefore G is pure. ∎

The theorem is stronger than practical reproducibility, but the latter follows immediately. If two executions use the same seed and the same fixed configuration, then by purity they produce the same output. Byte-level identity follows when the serializer is canonical.

7. Cross-Platform Reproducibility Evaluation

We evaluated the methodology by regenerating identical public and hidden suites on Linux, macOS, and Windows using CPython 3.10 through 3.13. For each environment we synthesized 240 public cases and 240 hidden cases with the same seed and configuration, then computed SHA-256 digests of the resulting line-delimited JSON files. We additionally recomputed expected_sha256 entries from the evaluator to ensure semantic agreement.

These results support the practical claim that same seed implies identical dataset output across the tested platforms. The finding depends on disciplined implementation choices. We use integer-valued seeds, deterministic JSON key ordering, stable formatting of identifiers, and no dependence on locale-specific formatting. We also avoid unordered containers in the output schema and avoid sources of randomness with platform-specific backends. When these conditions are met, reproducibility becomes an engineered invariant rather than an aspirational hope.

8. Implementation Considerations

Several low-level details determine whether a nominally deterministic design remains reproducible in deployed systems. First, every helper routine should receive rng explicitly rather than reading global state. This turns hidden side effects into visible data flow and simplifies code review. Second, identifiers should be formatted from deterministic fields only. In FableBreaker, case identifiers incorporate split, zero-padded index, and family name; none of these depend on machine state. Third, serialization must be canonical. Using json.dumps(record, sort_keys=True) and UTF-8 output ensures stable field ordering and encoding.

Fourth, evaluation of generated cases should be deterministic and preferably total over the intended subset of the language. The FableBreaker evaluator records both reference_steps and reference_max_depth, supplying an audit trail for the computational profile of each case. Fifth, benchmark maintainers should version not only code but also parameter files. A change from one decay schedule to another is a scientific change to the benchmark distribution, even if the generator entry point remains the same.

Finally, deterministic generation should be tested automatically. Unit tests can instantiate random.Random(823), synthesize a prefix of cases, and verify that expected digests and reference statistics are stable. Integration tests can regenerate full datasets and compare manifest hashes across CI environments.

9. Discussion

The seed-controlled methodology solves one important part of reproducibility, but not all of it. It guarantees exact corpus regeneration under fixed configuration; it does not by itself guarantee that benchmark operators disclose the relevant parameters, that candidate evaluators are deterministic, or that runtime measurements are variance-free. Those concerns require complementary governance and scoring procedures. Nevertheless, deterministic generation is foundational. Without it, all downstream controls operate on moving sand.

A second limitation concerns implementation portability. The purity theorem assumes a fixed pseudorandom algorithm and serializer. If a future runtime alters the generator's algorithmic contract, historical seeds may no longer replay identically. The remedy is straightforward: record interpreter versions in benchmark metadata and treat changes in core pseudorandom semantics as benchmark-version changes. For maximal long-term archival stability, projects may embed a self-contained pseudorandom engine rather than relying on a language runtime. FableBreaker's present design demonstrates that strong reproducibility is already attainable with standard library components when disciplined engineering constraints are enforced.

10. Conclusion

Deterministic test generation via seed-controlled stochastic AST synthesis reconciles two requirements that are often treated as oppositional: structural diversity and exact reproducibility. By making random.Random(seed) the sole source of stochasticity, parameterizing node and family distributions explicitly, controlling depth with hard limits and decay schedules, and serializing outputs canonically, FableBreaker obtains a benchmark generator that is both expressive and auditable. The same seed yields the same dataset; the result is not folklore but a theorem backed by implementation discipline and cross-platform replication. We argue that future benchmark publications should report generator seeds and parameter vectors alongside aggregate scores, thereby elevating reproducible test synthesis to a first-class scientific norm.
