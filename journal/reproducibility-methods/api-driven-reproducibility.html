API-Driven Reproducibility: HTTP Service Architectures for Benchmark Automation

Freddy Medina
ItsNotAI LABS Research Division
June 2026

Abstract

Reproducible benchmarks require not only deterministic computation but also standardized interfaces for invoking generation, scoring, and certification workflows. Command-line tools, while effective for manual operation, introduce variability through shell environments, working directory assumptions, and argument parsing differences across platforms. This paper presents an HTTP service architecture for FableBreaker that provides a versioned, self-documenting API for all benchmark operations. The service implements rate limiting, CORS support for browser-based tooling, structured error responses, request tracing, and backward-compatible route evolution. We describe the design principles—statelessness, idempotency of read operations, explicit versioning, and defense against abuse—and demonstrate that API-driven benchmarking achieves equivalent reproducibility to command-line invocation while enabling integration with CI/CD pipelines, web dashboards, and automated certification workflows without platform-specific scripting.

Keywords: HTTP API, reproducibility, benchmark automation, service architecture, rate limiting, versioned endpoints, CI/CD integration

1. Introduction

The standard interface for benchmark systems is the command-line tool. Users invoke a generator with specific arguments, pipe output to a scorer, and collect results from a report file. This workflow is reproducible in principle—given identical inputs, the same outputs are produced—but fragile in practice. Shell environments differ between platforms, relative path resolution depends on working directory, and argument parsing may vary between Python versions or OS configurations. More critically, command-line interfaces resist automation: integrating a benchmark into a CI/CD pipeline requires shell scripting that is itself a source of irreproducibility.

An HTTP service interface addresses these limitations by providing a platform-independent, self-documenting, and programmatically accessible interface to benchmark operations. Any HTTP client—regardless of language, platform, or environment—can invoke the same operations with identical semantics. The service encapsulates all path resolution, environment configuration, and subprocess management internally, exposing only the semantic parameters of each operation.

2. Design Principles

2.1 Versioned Routes

All API endpoints are prefixed with a version identifier (/api/v1/). Legacy routes without the version prefix remain functional for backward compatibility but are documented as deprecated. This enables the service to evolve its interface without breaking existing integrations.

2.2 Statelessness

Each request is self-contained. The service does not maintain session state between requests. This ensures that any request can be replayed independently for debugging or verification, and that multiple clients can interact with the service concurrently without interference.

2.3 Idempotency

Read operations (health, manifest, score, status) are idempotent: invoking them multiple times produces the same result (modulo timing measurements, which are inherently non-deterministic). Write operations (generate) are deterministic given the same seed parameters, making them effectively idempotent for reproducibility purposes.

2.4 Defense Against Abuse

Rate limiting prevents resource exhaustion from automated clients. The service enforces a configurable request-per-window limit per client IP address. Requests exceeding the limit receive a 429 response with retry information. This protects shared benchmark infrastructure from denial-of-service while remaining transparent to well-behaved clients.

3. Service Architecture

3.1 Endpoint Catalog

The FableBreaker service exposes the following versioned endpoints:

  • GET /api/v1/health — Service health with version and timestamp
  • GET /api/v1/manifest — Benchmark suite manifest
  • GET /api/v1/status — Available datasets, reports, and configuration
  • GET /api/v1/candidates — Registered candidate modules
  • GET /api/v1/families — Available adversarial families with descriptions
  • GET /api/v1/score — Score a candidate (query parameters)
  • POST /api/v1/score — Score a candidate (JSON body)
  • POST /api/v1/generate — Generate a dataset from seed parameters

3.2 Response Format

All responses are JSON with consistent structure. Successful responses include the requested data directly. Error responses include an error field with a machine-readable error code and a human-readable message. All responses include service version and request timestamp headers for audit trail purposes.

3.3 CORS Support

The service includes full CORS support (Access-Control-Allow-Origin, preflight OPTIONS handling) to enable browser-based dashboard and visualization tools to interact with the benchmark service directly without proxy infrastructure.

4. CI/CD Integration Patterns

The HTTP interface enables several automation patterns:

  • Pull request gates: A CI job starts the service, submits the candidate for scoring, and fails the build if certification is not achieved
  • Nightly certification: A scheduled workflow generates fresh hidden-seed datasets and scores all registered candidates, publishing results to a dashboard
  • Regression alerts: Per-family speedup comparisons between commits detect performance regressions on specific adversarial families
  • Multi-platform validation: The same HTTP requests can be issued from any platform, enabling cross-platform reproducibility verification

5. Conclusion

HTTP service architectures provide a reproducibility layer above the computational layer. While deterministic generation and hash verification ensure that results are correct, the service interface ensures that the process of obtaining those results is standardized, automated, and platform-independent. The FableBreaker service demonstrates that benchmark automation does not require sacrificing rigor: every service operation maps directly to a well-defined benchmark primitive with identical semantics to its command-line counterpart.
