Hash-Based Collision Resistance in Semantic Output Verification

Freddy Medina
ItsNotAI LABS Research Division
June 2026
FBJ-SP-2026-002

Abstract

Semantic verification at benchmark scale requires an observation function that is deterministic, portable across implementations, and strong enough to distinguish every value reachable in the benchmark language. FableBreaker adopts a hash-based strategy: evaluator outputs are first normalized by a canonical serialization procedure and then digested with SHA-256. This paper analyzes the technical reasons for that design. We formalize canonicalization as a recursively defined normalization map over the benchmark's output domain, prove its determinism and representation independence, and characterize the injectivity conditions under which equal digests witness equal semantic values. We show why SHA-256 is an appropriate cryptographic choice for this setting, quantify collision probabilities under realistic benchmark cardinalities, and explain the precise limits of the guarantee: collision resistance is not semantic equivalence, but when combined with a deterministic canonical form and a constrained value domain it becomes an excellent practical certificate. Particular attention is given to nested structural outputs, map-key ordering, the distinction between semantic collisions and cryptographic collisions, and the challenges that floating-point values would introduce were they to be admitted into the language. We compare hashing against alternative approaches such as SMT-based equivalence checking and differential testing, and argue that the benchmark's method achieves a favorable balance of rigor, computational cost, portability, and operational security. The analysis establishes that benchmark-scale semantic certification can rely on standard cryptography without confusing cryptographic evidence for a proof of language equality in the abstract.

Keywords: SHA-256, canonical serialization, collision resistance, semantic hashing, deterministic verification, benchmark security, output normalization, alternative certification methods

1. Introduction

Output comparison is deceptively difficult in heterogeneous evaluator benchmarks. Two implementations may compute the same semantic value while choosing different in-memory representations, different host-language container types, or different key orders when serializing dictionaries. Conversely, two implementations may accidentally produce byte-identical output for distinct semantic values if the comparison function is poorly designed. FableBreaker addresses this problem by separating semantic normalization from cryptographic commitment. Each candidate result is first converted into a canonical representation and only then hashed with SHA-256. This design lets the benchmark compare values, not accidents of representation.

The decision to hash rather than compare canonical objects directly has operational and methodological advantages. Hashes are fixed-size, easy to store in datasets, resistant to accidental truncation or whitespace perturbation, and suitable for publication in audit artifacts. More importantly, precomputed hashes allow hidden datasets to expose expected outputs without revealing the outputs themselves. A candidate evaluator must therefore match the semantic observations of the reference interpreter without being able to memorize a visible corpus of large canonical values.

However, a hash is not a proof by itself. Collision resistance is probabilistic, canonicalization may or may not be injective, and some semantic properties—such as equivalence on diverging or exceptional executions—cannot be captured by digest comparison alone. The aim of this paper is to delineate precisely what the FableBreaker digest certifies, why SHA-256 is appropriate for that purpose, and where the approach must be supplemented by language-theoretic reasoning.

2. The Digest Pipeline in FableBreaker

The repository implements the digest pipeline in two concise functions:

def canonical(value):
    if isinstance(value, bool): return value
    if isinstance(value, int): return value
    if isinstance(value, str): return value
    if isinstance(value, list): return [canonical(item) for item in value]
    if isinstance(value, tuple):
        if value[0] == "tag":
            return {"tag": value[1], "value": canonical(value[2])}
        if value[0] == "pair":
            return {"pair": [canonical(value[1]), canonical(value[2])]}
    if isinstance(value, dict):
        return {str(k): canonical(v) for k, v in sorted(value.items())}


def digest(value):
    payload = json.dumps(canonical(value), sort_keys=True, separators=(",", ":"))
    return hashlib.sha256(payload.encode("utf-8")).hexdigest()

This pipeline has three stages. First, a host-language value is mapped into a canonical structural normal form. Second, the normal form is serialized to JSON with a fixed key order and compact separators, producing a unique byte string for each canonical object. Third, the byte string is hashed with SHA-256 to obtain a 256-bit digest encoded as hexadecimal. Determinism must hold at every stage; otherwise hash equality would fail to coincide with semantic equality even before cryptography enters the picture.

2.1 Why SHA-256?

SHA-256 occupies a pragmatic sweet spot. It is standardized, ubiquitous in language runtimes and security reviews, fast enough for benchmark-scale use, and conservative by contemporary cryptographic standards. FableBreaker does not require exotic properties such as homomorphism or zero-knowledge. It requires a widely trusted, collision-resistant digest that can be recomputed by independent auditors on commodity tooling. SHA-256 satisfies those criteria with ample security margin.

Alternative choices were possible. SHA-1 is obsolete for collision resistance. SHA-512 is also secure but provides little practical benefit for the benchmark's scale while doubling raw digest size if represented in full hexadecimal. BLAKE3 offers excellent performance, but SHA-256 has broader institutional familiarity and therefore lowers the burden on reviewers who must validate the packet. The benchmark optimizes for auditability rather than peak hashing throughput.

3. Determinism of Canonical Serialization

3.1 Formal definition

Let Rep be the host-language representation domain admitted by the candidate contract, and let Can be the JSON-compatible algebra generated by booleans, integers, strings, arrays, and ordered maps. Canonicalization is a partial function can : Rep ⇀ Can. It is partial because unsupported outputs, such as arbitrary objects or floating-point values, are rejected rather than coerced. This partiality is a feature: silent coercion would weaken the correspondence between semantic values and their certificates.

Definition 1 (determinism). can is deterministic when for every input x in its domain there exists exactly one y such that can(x) = y. The FableBreaker implementation is deterministic by structural recursion. Each branch is syntax-directed, recursive calls are made on strictly smaller substructures, tuple tags are disjoint, and dictionary keys are sorted before reconstruction.

3.2 Representation independence

The evaluator benchmark allows candidate modules to return values that are not byte-identical to the reference interpreter's Python tuples. A pair may be returned as a tuple by the baseline evaluator and as a dictionary by a candidate, yet both should compare equal if they represent the same semantic pair. Canonicalization achieves this by choosing canonical constructors {"pair": [...]} and {"tag": ..., "value": ...} as representation-independent observations.

Proposition 2. If two host values encode the same FableBreaker semantic value, then their canonical forms are equal. The proof is by structural induction on the semantic value, observing that lists, dictionaries, tags, and pairs are all canonicalized recursively and that dictionary keys are normalized to strings and sorted. The proposition is what makes digest comparison meaningful across implementations written by different teams.

3.3 Nested structures

Nested outputs are common because repeated pair and tag constructions build deeply recursive values. Canonicalization must therefore be stable under depth. The recursive definition ensures that if each strict subvalue has a unique canonical image, then the enclosing value does as well. For example, the semantic value tag("A", pair(3, tag("B", 9))) canonically becomes

{"tag":"A","value":{"pair":[3,{"tag":"B","value":9}]}}

The serialization is unambiguous because constructor names occupy fixed keys, arrays preserve left-to-right order, and maps have deterministic key order. Without this discipline, structurally equal nested outputs could hash differently across runtimes.

4. Collision Probability Analysis

Cryptographic collision resistance means that it is computationally infeasible to find distinct payloads p1 and p2 such that SHA-256(p1) = SHA-256(p2). For random-oracle style estimates, the birthday bound gives an upper approximation of n2 / 2257 for observing at least one collision among n random digests.

These bounds are astronomically smaller than ordinary engineering failure rates. Put differently, benchmark-scale verification is dominated by semantic modeling mistakes, implementation bugs, or dataset leakage long before it is meaningfully threatened by accidental SHA-256 collisions. The only realistic collision concern would be an active adversary deliberately constructing a second-preimage attack, but no practical attack of that kind is known for SHA-256 within the benchmark threat model.

It is important, however, not to overstate the conclusion. Collision analysis says that distinct canonical payloads are extraordinarily unlikely to share a digest. It does not say that distinct semantic values cannot canonicalize to the same payload. That latter property depends on the injectivity of canonicalization, which is a language-design issue rather than a cryptographic one.

5. From Hash Equality to Semantic Equality

Let Val be the reachable semantic output domain of the FableBreaker evaluator: integers, pairs, and tags generated by the reference semantics. Define ser(v) = JSON(can(v)) with sorted keys and compact separators, and dig(v) = SHA-256(ser(v)). The logical path from digest equality to semantic equality has three links:

  1. Serialization determinism: equal canonical objects yield equal byte strings.
  2. Canonical injectivity on Val: equal canonical objects correspond to equal semantic values.
  3. Collision resistance: distinct byte strings are computationally infeasible to map to the same digest.

Theorem 3. For v1, v2 ∈ Val, if dig(v1) = dig(v2) and no SHA-256 collision has occurred, then v1 = v2. The proof is immediate from the three links above. This is the exact semantic meaning of a certified benchmark match.

The benchmark's phrasing must therefore be precise. Hash equality does not replace semantics; it packages semantic equality into a compact certificate after semantics has been normalized. The digest is a witness for the canonical observation, not a stand-alone logical oracle.

5.1 Digest function properties

The implementation inherits several useful properties from SHA-256 and the serialization layer:

  • Determinism: the same semantic output always yields the same hexadecimal digest.
  • Compactness: each certificate is 64 hexadecimal characters regardless of output size.
  • Compositionality through canonicalization: nested outputs are reduced recursively before hashing.
  • Audit portability: any verifier with JSON and SHA-256 support can recompute the digest.
  • Second-preimage resilience: practical attacks against benchmark-scale certificates are out of reach.

6. Limitations of Hash-Based Verification

6.1 Hash collisions versus semantic collisions

A cryptographic collision is a pair of distinct serialized payloads with the same digest. A semantic collision is a design failure in which distinct semantic outputs canonicalize to the same serialized payload. These are conceptually different failure modes. The former is a property of the hash family; the latter is a property of the observation function. FableBreaker's security argument depends on ruling out semantic collisions by construction and then relying on SHA-256 to make cryptographic collisions negligible.

For example, if canonicalization had encoded both pairs and two-element lists as bare JSON arrays, then the semantic values pair(1,2) and [1,2] would collide before hashing. The actual implementation avoids this by wrapping pairs as {"pair": [...]}. Likewise, tags are distinguished by a dedicated tag key. Constructor names are part of the semantics, not cosmetic labels.

6.2 Failures and non-values

The digest pipeline does not directly represent exceptional behavior. If a candidate raises an exception or returns an unsupported value such as a float, canonicalization or evaluation fails and the scorer records a failure instead of a digest. Therefore hash-based verification applies to successful evaluations only. Semantic equivalence over errors still matters, but it is enforced procedurally—through failing the benchmark on exceptions—rather than through a separate error-hashing scheme.

6.3 Finite evidence versus universal proofs

Another limitation is epistemic. Matching digests on a finite dataset proves equality only on that dataset. It does not establish universal program equivalence for all possible closed expressions. Hidden seeds expand the tested domain and reduce gaming, but they do not solve the undecidable problem of general evaluator equivalence. Benchmark hashes are strong certificates of empirical semantic agreement, not substitutes for a mechanized proof assistant.

7. Floating-Point Determinism Challenges

The current FableBreaker language avoids floating-point numbers entirely. This omission is strategic. Floating-point semantics is notoriously vulnerable to cross-platform nondeterminism: extended precision registers, library-specific transcendental functions, quiet versus signaling NaNs, and multiple textual renderings of the same binary value can all destabilize hashing if not normalized with extreme care.

If the language were extended with floats, the present canonicalization function would be inadequate because Python float values are currently outside its domain. A sound extension would need to specify at least: (i) whether values are modeled as IEEE-754 bit patterns or as real numbers with rounding; (ii) how -0.0, NaNs, and infinities are represented; and (iii) whether serialization uses hexadecimal bit-pattern notation rather than decimal printing. Without such a policy, semantically identical computations on two runtimes could yield distinct hashes.

This observation clarifies an important design principle: a hash function can only be as deterministic as the serialization of the underlying domain. By restricting the benchmark to integers and structural data, FableBreaker achieves much stronger portability than many floating-point-heavy evaluation tasks.

SMT solvers are especially attractive for algebraic rewrite validation. They can prove, for example, that add(lit(a), lit(b)) and lit(a+b) are equivalent over integers. But benchmark certification operates on whole evaluators, not isolated expressions. Encoding the entire operational semantics of match, dup, and repeat into an SMT-friendly form is possible only for narrow fragments. Hashing therefore plays a different role: it provides low-cost, case-by-case certification after semantic normalization.

Differential testing alone also falls short because it lacks a portable observation format. If one evaluator returns tuples and another returns dictionaries, naive equality may fail even when the semantic value is identical. Canonical hashing removes this source of noise and allows hidden datasets to publish expectations without revealing full outputs.

9. Practical Security Margins for Benchmark-Scale Verification

Benchmark verification operates at a scale where operational practicality matters. A typical FableBreaker dataset contains hundreds of cases, each of which may be rerun multiple times for timing statistics. Hashing adds negligible overhead compared with evaluation, especially for structurally recursive outputs of modest size. More importantly, the security margin provided by SHA-256 far exceeds what is needed. Even if one imagines millions of benchmark executions over the lifetime of the repository, the accidental-collision probability remains so small as to be irrelevant for scientific interpretation.

Security margin should also be understood socially. Reviewers, downstream users, and auditors are more likely to trust a benchmark protocol built from widely understood primitives than one built from bespoke serialization code alone. By basing certification on deterministic JSON serialization and a standard hash, FableBreaker reduces ambiguity in audit conversations. The evidence packet can state a digest, and any independent party can reproduce it from the canonical output with commodity tools.

Finally, practical security includes resistance to benchmark gaming. Publishing only expected digests for hidden cases leaks far less information than publishing full canonical outputs. A candidate evaluator cannot trivially reverse SHA-256 to recover the hidden values, which helps maintain the integrity of withheld workloads. The digest thus functions both as an equality certificate and as a minimal-disclosure commitment.

10. Conclusion

Hash-based verification in FableBreaker is neither cryptographic theater nor a magical substitute for semantics. It is a disciplined composition of three ingredients: a deterministic canonicalization function, a constrained semantic output domain, and a collision-resistant standard digest. Within that composition, SHA-256 is an excellent choice. It yields compact, portable certificates; supports hidden-output benchmarking; and provides security margins that overwhelm realistic benchmark scales. Its limitations are equally clear: it does not prove universal evaluator equivalence, it does not directly encode exceptional behavior, and it would need extension before admitting floating-point semantics. Those limitations are acceptable because the benchmark uses hashing as one layer in a broader semantics-first verification design.
