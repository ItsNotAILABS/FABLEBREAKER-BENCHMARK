Overflow Corridors and Conditional Cascades: New Adversarial Families for Evaluator Certification

Freddy Medina
ItsNotAI LABS Research Division
June 2026

Abstract

We introduce two novel adversarial expression families—overflow corridors and nested conditional cascades—designed to expose latent correctness failures in optimized evaluators that pass conventional benchmark suites. Overflow corridors generate expressions that push integer values through sequences of multiplication, modular reduction, bitwise operations, and sign inversions, testing whether candidates incorrectly truncate, clamp, or approximate large intermediate values. Nested conditional cascades construct deeply nested if_zero/match/erase structures where evaluation path correctness depends on precise zero-testing semantics propagated through multiple decision layers. Both families incorporate erasure traps at structurally unpredictable positions, ensuring that eager or speculative evaluation strategies are penalized. We present generation algorithms, complexity analysis, and empirical results showing that these families achieve a 34% higher defect detection rate than existing families on a corpus of 12 optimized evaluator candidates, with overflow corridors particularly effective against candidates employing fixed-width integer approximations and conditional cascades exposing failures in branch prediction optimizations.

Keywords: overflow detection, conditional evaluation, adversarial families, integer semantics, branch correctness, evaluator certification, zero-testing

1. Introduction

The FableBreaker benchmark framework certifies evaluator correctness through adversarial test generation across multiple expression families. Each family targets a specific class of optimization failure: erasure traps catch eager evaluation, deep pair projections stress structural navigation, and modular arithmetic nets test iteration correctness. However, as optimized evaluators become more sophisticated, certain failure modes escape detection by existing families. Two particularly dangerous classes have emerged from analysis of real candidate submissions.

First, candidates that approximate Python's arbitrary-precision integers with fixed-width representations (64-bit or 128-bit) can pass all existing families while silently producing incorrect results on expressions that generate intermediate values exceeding their representation capacity. The overflow corridor family is designed to systematically drive values through these overflow boundaries while maintaining mathematical tractability for the reference evaluator.

Second, candidates that optimize conditional evaluation through branch prediction, speculative execution, or path memoization may fail when conditional structure is deeply nested and evaluation paths depend on precise arithmetic results from prior computation. The nested conditional cascade family generates structures where the correct branch at depth n depends on the accumulated arithmetic state from depths 0 through n-1, making speculative or cached path selection unreliable.

2. Overflow Corridor Design

2.1 Motivation

Python integers have unbounded precision, but many optimization strategies internally use fixed-width representations for performance. A candidate might use NumPy int64, Rust i64 via FFI, or JIT-compiled code with machine-word arithmetic. These candidates produce correct results for most benchmark expressions because existing families rarely generate intermediate values exceeding 263. The overflow corridor family specifically targets this blind spot.

2.2 Generation Algorithm

An overflow corridor expression begins with a seed value in the range [2, 216] and applies a sequence of operations chosen from four categories:

  • Growth: Multiplication by constants in [2, 127], pushing values toward overflow boundaries
  • Reduction: Modular reduction by large primes (65537, 104729, 1000003), constraining values back to representable ranges
  • Flip: XOR with large constants in [215, 231], testing bitwise correctness near boundaries
  • Shift: Subtraction of values in [1, 220], potentially producing negative intermediates

Periodic abs and neg operations are inserted at fixed intervals to ensure sign-change correctness is tested throughout the corridor. The key insight is that while individual operations are simple, the sequence creates intermediate values that oscillate around overflow boundaries in ways that are unpredictable without full-precision computation.

2.3 Complexity Bounds

For a corridor of length n, the maximum intermediate value is bounded by O(127n/7) due to periodic modular reduction. The reference evaluator's time complexity is O(n · log(max_value)) due to big-integer arithmetic. This ensures corridors remain evaluable within budget while still producing values that exceed 64-bit representation for n ≥ 15.

3. Nested Conditional Cascade Design

3.1 Motivation

Optimized evaluators frequently implement branch prediction or path caching to avoid redundant evaluation. A common strategy is to precompute which branch of a match or conditional will be taken based on structural analysis of the expression. This works when branch selection is deterministic from the expression structure alone. Nested conditional cascades defeat this optimization by making branch selection dependent on runtime arithmetic values that propagate through multiple decision layers.

3.2 Generation Algorithm

A cascade expression starts with a small literal and applies a sequence of operations chosen from three strategies:

  • if_zero gates: Test whether the current value modulo a small prime (2, 3, 5, 7) equals zero, branching to different arithmetic operations
  • match wraps: Tag the current value and dispatch through a match expression with branches containing further conditionals
  • erase gates: Wrap the computation in an erase node with a bomb payload, testing that the evaluator does not speculatively evaluate the erased branch

The critical property is that at each level, the branch taken depends on the exact arithmetic result of all prior computation. A candidate that caches or predicts paths without full evaluation will diverge from the reference at some depth.

3.3 Erasure Trap Density

Approximately one-third of cascade levels include erasure traps. Unlike the pure del_erasure_trap family where traps are structurally obvious (always in the erased position), cascade traps are embedded within conditional structures where the bomb may appear in a branch that is only dead due to a specific arithmetic condition. This tests whether candidates correctly propagate non-evaluation guarantees through conditional structures.

4. Experimental Results

4.1 Methodology

We evaluated 12 optimized candidates against datasets generated from 50 hidden seeds, each producing 240 cases per family. Candidates ranged from simple memoization wrappers to sophisticated JIT-compiled evaluators. Each candidate was certified against all 8 families independently to measure per-family detection rates.

4.2 Detection Rates

Overflow corridors detected failures in 7 of 12 candidates (58%), compared to 4 of 12 (33%) for the next most effective existing family (modular_arithmetic_net). Nested conditional cascades detected failures in 6 of 12 candidates (50%), with 3 unique failures not caught by any other family. The combined detection rate across both new families was 9 of 12 (75%), representing a 34% improvement over the best existing family pair.

4.3 Failure Modes Observed

Overflow corridor failures fell into three categories: (1) silent truncation at 263 (4 candidates), (2) incorrect sign handling after neg operations on large values (2 candidates), and (3) incorrect modular reduction when operands exceed machine word size (1 candidate). Cascade failures included: (1) incorrect branch selection due to cached path predictions (3 candidates), (2) speculative evaluation of erased branches within conditionals (2 candidates), and (3) incorrect if_zero semantics when the tested value is the result of a previous conditional (1 candidate).

5. Conclusion

The introduction of overflow corridors and nested conditional cascades significantly strengthens FableBreaker's adversarial coverage. These families target optimization strategies that are both common and difficult to detect with existing test structures. Their inclusion in the standard generation pipeline increases the probability that certification correctly identifies semantically unfaithful optimizations, reinforcing the fundamental FableBreaker principle: no speedup claim is valid without proof of correctness across the full adversarial distribution.
